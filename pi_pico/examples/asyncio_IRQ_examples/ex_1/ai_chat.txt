To determine the ideal size for your ring buffer, you need to balance system memory against the burst rate of your interrupts.
There is no "magic number," but you can calculate a safe estimate using three key factors:
1. The "Burst Capacity" Calculation
The buffer's main job is to hold data during the time your asyncio task is "busy" or "sleeping."

* Max Interrupt Rate: How many IRQs fire per second? (e.g., 100/sec)
* Max Latency: What is the longest time your asyncio loop might be busy before it returns to await signal.wait()? (e.g., 0.5 seconds if you have a slow task).
* Formula: (Rate * Latency) * Safety Margin = Buffer Size
* Example: (100 IRQs/sec * 0.5 sec) * 2 (safety) = 100 slots.

2. Empirical Testing (The Best Method)
Since asyncio timing can be unpredictable, the most reliable way is to monitor the "High Water Mark" during development. Add a "peak usage" variable to your RingBuffer class:

class RingBuffer:
    def __init__(self, size):
        self.size = size
        self.buffer = [0] * size
        self.count = 0
        self.peak = 0  # <--- Track the maximum ever reached

    def put(self, val):
        if self.count == self.size: return False
        # ... (rest of put logic)
        self.count += 1
        if self.count > self.peak:
            self.peak = self.count # Update peak
        return True

Run your system under the "worst-case" stress (e.g., trigger all 3 IRQs as fast as possible). If peak reaches your size, your buffer is too small. If peak only reaches 10% of size, you are wasting RAM.
3. Memory Constraints
On MicroPython boards (like the ESP32 or RP2040), RAM is limited.

* Small Ints: A list of 100 small integers is very cheap (a few hundred bytes).
* Large Buffers: If you need thousands of slots, use array.array('I', [0]*1000) instead of a list. array.array is much more memory-efficient because it stores raw bytes rather than Python object pointers.

4. The "Overflow" Strategy
Decide what happens if the buffer does fill up. You have two choices:

* Reject New: Keep the old data, ignore the new (current put logic).
* Overwrite Old: Drop the oldest unread item to make room for the new one. This is usually preferred for real-time sensors where the newest data is the most relevant.

In MicroPython, micropython.schedule and asyncio are separate systems that work together to bridge the gap between high-priority, restrictive hardware interrupts and your standard application code. [1, 2] 
How They Interact

* Execution Order: micropython.schedule has higher priority than asyncio. When an IRQ calls schedule(), the scheduled function is placed in a queue. MicroPython will execute this function as soon as it returns to the "main" context—typically between two bytecode instructions—even if an asyncio task is currently running.
* Context Bridge: The primary interaction is that schedule() allows you to safely set a ThreadSafeFlag or update shared data. Because asyncio is cooperative, it cannot "know" an interrupt happened until it yields control (using await). schedule() ensures the data is moved into a safe place (like a list) so that when the asyncio loop next resumes, the data is ready for the task to process.
* Memory Allocation: asyncio tasks and coroutines can allocate memory freely. Hardware IRQs cannot. micropython.schedule acts as the "middleman" that runs in a context where memory allocation is allowed, making it the perfect place to append() to lists or set() flags for your tasks. [2, 3, 4, 5, 6, 7, 8] 

Key Performance Note
While schedule() is fast, there is a variable amount of latency between when the IRQ fires and when your asyncio task actually processes it. This is because the asyncio task must wait for the currently running task to hit an await statement before it can wake up and check the flag. [4, 5, 9] 

| Feature | Hardware IRQ | micropython.schedule | asyncio Task |
|---|---|---|---|
| Priority | Absolute (Preemptive) | High (Interrupts main) | Low (Cooperative) |
| Memory Allocation | Forbidden | Allowed | Allowed |
| Best For | Toggling a Pin | Moving data to a buffer | Business logic / IO |

import asyncioimport micropythonimport machineimport time
# --- Configuration & Buffering ---# Pre-allocate a list to act as our data bridge# micropython.schedule handles the memory safety between IRQ and Taskdata_queue = []data_signal = asyncio.ThreadSafeFlag()
# --- IRQ & Scheduler Logic ---
def scheduled_handler(val):
    """
    Runs in the 'main' context.
    Safely appends data and signals the asyncio event loop.
    """
    data_queue.append(val)
    data_signal.set()
def irq_handler(pin):
    """
    Hardware IRQ: Runs instantly. 
    Cannot allocate memory; must use schedule().
    """
    # We pass the pin ID (or any small int) to the scheduler
    # Assuming pin IDs 0, 1, and 2 for this example
    pin_num = 0 
    micropython.schedule(scheduled_handler, pin_num)
# Setup simulated IRQs (Change pins to match your hardware)# For demo purposes, we'll just define the handler.# pin0 = machine.Pin(0, machine.Pin.IN, machine.Pin.PULL_UP)# pin0.irq(trigger=machine.Pin.IRQ_FALLING, handler=irq_handler)
# --- Asyncio Tasks ---
async def data_processor():
    """
    Consumes data whenever the signal flag is set.
    """
    print("[Processor] Started and waiting for IRQ data...")
    while True:
        # 1. Wait for the signal (Non-blocking to the rest of the loop)
        await data_signal.wait()
        
        # 2. Process all items currently in the queue
        while data_queue:
            item = data_queue.pop(0)
            print(f"[Processor] Handling data item: {item}")
        
        # ThreadSafeFlag.wait() automatically clears itself in MicroPython
async def background_monitor():
    """
    Illustrates that the loop is NOT blocked. 
    This runs even if no IRQs are firing.
    """
    count = 0
    while True:
        print(f"[Monitor] Heartbeat {count}...")
        count += 1
        await asyncio.sleep(5) # Give other tasks time to run
async def main():
    # Start both tasks concurrently
    print("--- System Starting ---")
    
    # Create the tasks
    processor = asyncio.create_task(data_processor())
    monitor = asyncio.create_task(background_monitor())
    
    # Keep main alive
    while True:
        await asyncio.sleep(1)
# --- Execution ---try:
    asyncio.run(main())except KeyboardInterrupt:
    print("System Stopped")

Key Highlights of this Program:

   1. ThreadSafeFlag: This is the "bridge" that allows the data_processor to sleep efficiently until an IRQ happens.
   2. micropython.schedule: This ensures that even if the IRQ fires while the list is full, the .append() operation happens in a memory-safe context.
   3. data_queue.pop(0): Inside the while data_queue loop, we handle multiple interrupts that might have occurred back-to-back before the asyncio task had a chance to wake up.
   4. background_monitor: This task proves the system is non-blocking. It continues to print "Heartbeat" every 5 seconds regardless of whether data is coming in from the IRQs.
