In MicroPython, micropython.schedule and asyncio are separate systems that work together to bridge the gap between high-priority, restrictive hardware interrupts and your standard application code. [1, 2] 
How They Interact

* Execution Order: micropython.schedule has higher priority than asyncio. When an IRQ calls schedule(), the scheduled function is placed in a queue. MicroPython will execute this function as soon as it returns to the "main" context—typically between two bytecode instructions—even if an asyncio task is currently running.
* Context Bridge: The primary interaction is that schedule() allows you to safely set a ThreadSafeFlag or update shared data. Because asyncio is cooperative, it cannot "know" an interrupt happened until it yields control (using await). schedule() ensures the data is moved into a safe place (like a list) so that when the asyncio loop next resumes, the data is ready for the task to process.
* Memory Allocation: asyncio tasks and coroutines can allocate memory freely. Hardware IRQs cannot. micropython.schedule acts as the "middleman" that runs in a context where memory allocation is allowed, making it the perfect place to append() to lists or set() flags for your tasks. [2, 3, 4, 5, 6, 7, 8] 

Key Performance Note
While schedule() is fast, there is a variable amount of latency between when the IRQ fires and when your asyncio task actually processes it. This is because the asyncio task must wait for the currently running task to hit an await statement before it can wake up and check the flag. [4, 5, 9] 

| Feature | Hardware IRQ | micropython.schedule | asyncio Task |
|---|---|---|---|
| Priority | Absolute (Preemptive) | High (Interrupts main) | Low (Cooperative) |
| Memory Allocation | Forbidden | Allowed | Allowed |
| Best For | Toggling a Pin | Moving data to a buffer | Business logic / IO |
